АҢДАТПА

Дипломдық жұмыс «Stolyarniy dvor» ағаш өңдеу цехіне арналған микросервистік веб-қосымшаны әзірлеу мәселесіне арналған. Жұмыста өндіріс саласындағы тапсырыс қабылдау мен басқару процесінің проблемалары зерделенді. Жүйе React (Vite), Node.js, Express.js, Prisma ORM және PostgreSQL технологияларын қолдана отырып, микросервистік архитектура (api-gateway, orders-service, products-service) негізінде жобаланды. Әкімшілік панель арқылы (dashboard) тапсырыстар мен пайда динамикасын графиктермен бейнелеу, Excel-де есептер генерациялау мүмкіндігі іске асырылды. Сонымен қатар жаңа тапсырыстарды Telegram Bot API арқылы жіберу интеграцияланды. Версияларды басқару Git/GitHub арқылы жүзеге асырылып, жоба Docker Compose және Nginx арқылы Render бұлттық платформасында орналастырылды.

АННОТАЦИЯ

Дипломная работа посвящена разработке микросервисного веб-приложения для деревообрабатывающего цеха «Stolyarniy dvor». Исследованы проблемы процессов приема заказов и управления на производстве. Система спроектирована на основе микросервисной архитектуры (api-gateway, orders-service, products-service) с использованием React (Vite), Node.js, Express.js, Prisma ORM и PostgreSQL. Реализована функциональная панель администратора (dashboard) с отображением аналитики по заказам и выручке с помощью графиков и генерацией отчетов в Excel. Интегрировано автоматическое уведомление о поступлении новых заказов через Telegram Bot API. Контроль версий осуществлялся с помощью Git/GitHub, развертывание выполнено с помощью Docker Compose и Nginx на облачной платформе Render.

ANNOTATION

The graduation project is devoted to the development of a microservice web application for the woodworking workshop «Stolyarniy dvor». The problems of order acceptance and management processes in production have been studied. The system is designed based on a microservice architecture (api-gateway, orders-service, products-service) using React (Vite), Node.js, Express.js, Prisma ORM and PostgreSQL. A functional administrator panel (dashboard) has been implemented to display analytics on orders and revenue using charts and generate Excel reports. Automatic notification of new orders via the Telegram Bot API is integrated. Version control was managed via Git/GitHub, and deployment was performed using Docker Compose and Nginx on the Render cloud platform.

КІРІСПЕ

Жұмыстың өзектілігі. Қазіргі кезде технологиялық прогресс пен цифрландыру экономиканың барлық салаларына, соның ішінде өндірістік және қызмет көрсету секторларына терең енді. Шағын және орта бизнес субъектілері үшін тұтынушыларға жоғары сапалы қызмет көрсету, ішкі бизнес-процестерді тиімді басқару, қаржылық нәтижелерді нақты уақыт режимінде қадағалау маңызды рөл атқарады. Ағаш өңдеу цехтары мен столярлық өндіріс орындары да бұл үрдістен тыс қалмауы тиіс. Кез келген заманауи өндіріс клиенттік сұраныстарды жылдам өңдеуді, менеджмент құралдары арқылы тауар айналымын оңтайландыруды, сондай-ақ статистикалық мәліметтерге сүйене отырып шешімдер қабылдауды қажет етеді [1].
Бірақ тәжірибе көрсеткендей, көптеген столярлық цехтар әлі күнге дейін тапсырыстар мен қаражатты есепке алуды қағаз дәптерлерде, жай Excel кестелерінде немесе қарапайым мессенджерлерде жүргізеді. Бұл деректердің жоғалу қаупін тудырады, статистикалық және қаржылық талдау жасауды қиындатады, сонымен бірге клиенттерге қызмет көрсету жылдамдығын төмендетеді [2]. Осы тұрғыда «Stolyarniy dvor» цехы үшін микросервистік архитектураға негізделген, тапсырыстарды, клиенттер базасын және қаржылық аналитиканы басқаратын толыққанды корпоративтік веб-қосымша әзірлеу өзекті міндетке айналып отыр.
Бүгінгі таңда веб-қосымшаларды және электрондық коммерция жүйелерін әзірлеуге арналған ғылыми жұмыстар мен зерттеулердің кең ауқымы қалыптасқан. Соның ішінде, 2018 жылы Google компаниясының мамандары жариялаған зерттеулерде Progressive Web Application (PWA) технологияларының веб-жүйелер өнімділігі мен қолжетімділігін арттырудағы рөлі қарастырылған [2]. Сонымен қатар, 2021 жылы Amazon компаниясының электрондық коммерция саласындағы тәжірибелері онлайн тапсырыс беру жүйелерінің масштабталуы, сенімділігі және пайдаланушыға ыңғайлылығы бағытында маңызды үлгі ретінде сипатталған [3]. Аталған зерттеулерде микросервистік архитектура, бұлттық платформалар және REST API технологияларын қолданудың тиімділігі жан-жақты талданады.
Тақырыптың зерттелу деңгейі. Ағаш өңдеу өндірісін автоматтандыру және веб-қосымшалар негізінде басқару мәселелері бірқатар ғылыми және қолданбалы зерттеулерде қарастырылған. Бұл жұмыстарда өндірістік процестерді цифрландыру, деректерді өңдеу, тапсырыстарды басқару және ресурстарды тиімді пайдалану бағыттары зерттеледі. Зерттеулер нәтижесінде веб-технологиялардың икемділігі мен қолжетімділігі дәлелденгенімен, ағаш өңдеу саласының ерекшеліктерін толық ескеретін бейімделген бағдарламалық шешімдер жеткіліксіз деңгейде қарастырылған, бұл аталған бағытта жаңа жүйе әзірлеудің өзектілігін айқындайды.
Зерттеу нысаны-«Stolyarniy dvor» столярлық цехының ақпараттық және бизнес-процестері.
Зерттеу пәні - микросервистік веб-қосымшаларды жобалау, әзірлеу, тестілеу және бұлтта орналастырудың заманауи әдістері мен құралдары. 
Жұмыстың мақсаты; Stolyarniy dvor цехіне арналған микросервистік веб-қосымшаны әзірлеу.
Осы мақсатқа жету үшін төмендегідей міндеттер қойылды:
-пәндік аумақты талдау және өндірістік процестерді цифрландыру қажеттілігін негіздеу;
-жүйенің архитектурасын таңдау және негіздеу;
-деректер қорын және жүйенің құрылымдық компоненттерін жобалау;
-бағдарламалық жүйені жүзеге асыру;
-жүйенің жұмысын тестілеу және нәтижелерін талдау.
Теориялық маңыздылығы. Веб-қосымшаларды әзірлеу барысында қолданылатын жобалау тәсілдері мен құрылымдық шешімдерді жүйелеуде көрінеді. Зерттеу нәтижелері ақпараттық жүйелерді әзірлеумен айналысатын мамандарға, сондай-ақ студенттерге бағдарламалық жүйелерді жобалау, деректер қорын ұйымдастыру және жүйе компоненттерінің өзара әрекеттесуін құру бойынша әдістемелік негіз ретінде қолдануға мүмкіндік береді.
Практикалық маңыздылығы. Әзірленген веб-қосымша «Stolyarniy dvor» цехында тапсырыстар циклін басынан аяғына дейін қадағалаудың негізгі құралына айнала алады. Интерактивті графиктер мен аналитикалық блоктар кәсіпорын иелеріне бизнестің кірістілігін, сатылым өсімін бақылап, дұрыс коммерциялық шешім қабылдауға мүмкіндік береді.
Дипломдық жұмыс кіріспеден, үш тараудан, қорытындыдан, пайдаланылған әдебиеттер тізімінен және қосымшалардан тұрады.
Бірінші тарауда өндірістік процестер мен тиісті проблемаларға талдау жасалды, аналогтар қаралды, сондай-ақ жүйеге қойылатын функционалдық және техникалық талаптар тұжырымдалды.
Екінші тарауда жобалау негіздері қамтылды: микросервистік архитектураның схемасы, технологиялық стектің (React, Node.js, Prisma, PostgreSQL, Docker) негіздемесі, API эндпоинттардың маршрутизациясы, мәліметтер қорының құрылымы, және Telegram Bot пен Git/GitHub пайдалану логикасы суреттелді.
Үшінші тарауда қосымшаның тікелей іске асырылу барысы баяндалды: графикалық интерфейстің әзірленуі (әкімші панелі, аналитика, Excel генерациясы), тестілеудің нәтижелері, қауіпсіздік шаралары және бұлтты платформада (Render) орналастыру кезеңдері.








1 ПӘНДІК АУМАҚТЫ ТАЛДАУ ЖӘНЕ ЕСЕПТІҢ ҚОЙЫЛЫМЫ


1.1 Ағаш өңдеу өндірісі және оларға арналған қосымшаларды талдау

«Stolyarniy dvor» ағаш өңдеу цехы ағаштан түрлі бұйымдар мен жиһаз элементтерін өндірумен айналысатын кәсіпорын. Оның негізгі клиенттік аудиториясы-жеке тұтынушылар және ішкі құрылысқа қажетті материалдарға тапсырыс беретін ұсақ көтерме сатып алушылар. Кәсіпорынның дамуы мен табыстылығы өндірілетін өнімнің сапасына ғана емес, сонымен бірге клиенттерден түсетін тапсырыстарды есепке алу, олардың орындалу барысын қадағалау, және жалпы түсім ағынын аналитикалық тұрғыдан сараптау процестерінің тиімділігіне тікелей байланысты болады.
Кәсіпорындағы басқару процестерінің ағымдағы ахуалы келесідей: тапсырыстар бойынша мәліметтерді қабылдау клиенттермен тікелей кездесіп сөйлесу немесе телефон арқылы, ал кейде WhatsApp мессенджері арқылы жүзеге асырылады. Менеджер әр тапсырыстың құрылымын, габариттерін және оның құнын қағаз блокноттарға немесе оффлайн режиміндегі электрондық кестелерге жазып алады. Осындай әдіс арқылы жинақталған ақпарат негізінде кәсіпорынның сату динамикасы мен қаржылық жағдайын бағалау өте күрделі [2].
Мұндай жұмыс тәртібінің негізгі проблемалық аспектілері мыналар:
Біріншіден, мәліметтердің фрагментациялануы. Ақпарат әртүрлі жерде (мессенджер чаттары, дәптер, Excel-файлдар) шашыраңқы сақталатындықтан, қай клиенттің тапсырысы қай өндіріс сатысында тұрғанын біріктірілген жүйеде көру мүмкін емес.
Екіншіден, қаржылық бақылау мен аналитиканың жоқтығы. Кәсіпорын айлық таза пайданы, орташа чекті, жаңа тапсырыстардың өсімін нақты және автоматты түрде есептей алмайды. Бұл деректерді жинау үшін менеджмент көп уақытты қолмен санауға жұмсайды. Ал тарихи деректерді алдыңғы аймен немесе тоқсанмен салыстырып, өсім немесе құлдырау пайызын шығару мүлдем қиынға соғады.
Үшіншіден, коммуникациялық олқылықтар. Клиент жаңа тапсырыс берген кезде ақпарат шеберхана қызметкерлеріне кешігіп жетуі мүмкін. Ауызша берілген ақпараттың жоғалу қаупі адам факторына байланысты едәуір артады.
Веб-технологиялар осы мәселелерге нақты жауап бере алады. Клиент тауарды өзі таңдап, тапсырысты интернет арқылы берсе, ал цех басшылығы барлығын бір орталықтандырылған панельден (dashboard) бақыласа - процестер едәуір реттеледі [3]. Жүйеде тапсырыстың статусы, төлем шарты және клиент деректері дерекқорда жинақталады. Аналитика автоматты есептеледі, ал Telegram Bot арқылы менеджер жаңа тапсырыс туралы лезде хабардар болады.
Кәсіпорын есепшісіне қажетті кезеңдік (ай, тоқсан, жыл) есептерді Excel форматында бір батырмамен жүктеп алу мүмкіндігі де жүйеге енгізілген.

Өндіріс пен сатуды басқаруға арналған нарықта бірқатар электрондық коммерция және CRM жүйелердің дайын шешімдері бар. Олардың функционалдық көлемі, архитектуралық модельдері және шағын өндірістік цехтар үшін жарамдылығы әртүрлі.
Shopify. Әлемдегі ең ірі электрондық коммерция платформаларының бірі. Тауарларды қою, себет, төлемдер жүйесі, және аналитикалық панелі жақсы дамыған [4]. Бірақ Shopify шешімдері бұлттық абоненттік жүйе (SaaS) моделімен жұмыс істейді. Әр қолданушы сайынғы абоненттік төлем мен комиссиялар шағын қазақстандық цех үшін тиімсіз болып табылады. Оған қоса Telegram-ға тікелей тапсырыстар жіберу сияқты жергілікті микро-функциялар базалық комплектацияда жоқ, ол үшін қосымша ақылы плагиндер орнату қажет.
AmoCRM / Bitrix24. Бұл құралдар негізінен сату воронкаларын басқаруға және клиенттермен қарым-қатынасты (CRM) жүргізуге бағытталған монолитті жүйелер. Аналитика да, есептер де бар. Бірақ олар веб-витрина (фронтенд) емес. Яғни, оларды пайдалану үшін алдымен жеке веб-сайт жасап, содан кейін күрделі интеграция жасау керек. Өзіндік микросервис архитектурасы мен теңшелетін функциялар арқылы икемдеуге де мүмкіндік жоқ [3].
Жергілікті кішігірім конструкторлар (Tilda, Wix). Көптеген цехтар электронды коммерцияға көшу үшін алғашында Tilda платформасын пайдаланады. Tilda тапсырыстарды қабылдауға арналған формаларды ұсынады және оны Telegram-ға байлауға болады. Алайда, мұнда құндылық тек ақпаратты визуалды көрсетуде ғана. Онда күрделі дашборд қалыптастыру, кезеңдермен салыстырмалы графиктер сызу, таңдамалы дата бойынша толық Excel-генерирлеу және тауар мен тапсырыс мәліметтерін өзара тәуелсіз реляциялық дерекқорларда басқару мүмкін емес.
«Stolyarniy dvor» үшін ұсынылған жеке микросервистік жүйе. Бұл жоба толығымен тапсырыс берушінің бизнес-қажеттіліктеріне бейімделіп жазылған клиент-серверлік қолданба болып табылады. Микросервистік архитектураның арқасында (products-service, orders-service) жүйе модульді түрде жұмыс істейді [4]. Егер болашақта өнімдерді басқару қызметі кеңейтілсе немесе жаңа төлем жүйесі қосылса, қосымшаны толық тоқтатпай, тек бір сервисті ғана жаңартуға болады.
Жүйеге кіріктірілген әкімшілік панель өндіріске қажетті төрт негізгі метриканы есептейді: жалпы тапсырыстар саны, жаңа тапсырыстар саны, жалпы выручка, орташа чек. Бұл көрсеткіштер React арқылы графиктермен динамикалық бейнеленеді және таңдалған уақыт аралығына (күн, апта, тоқсан) қарай өзгереді.

1.2 Ағаш өңдеу өндірісіндегі негізгі процестерді талдау және өзекті мәселелерін анықтау

«Stolyarniy dvor» цехының жұмыс барысын мұқият бақылау нәтижесінде бірнеше тұрақты ұйымдастырушылық проблема анықталды. Олар кішігірім көрінгенімен, уақыт өте келе кәсіпорынның өсуіне кедергі болады.
Біріншіден, тапсырысты қабылдау процесі стандартталмаған. Бір клиент WhatsApp арқылы хабарласса, екіншісі телефон соғады, үшіншісі тікелей келеді. Нәтижесінде мәліметтер шашыраңқы болып, жоғалу қаупі артады.
Екіншіден, өнімдер каталогы цифрлық түрде жоқ. Клиент бағаны, өлшемді, материалды алдын ала көре алмайды - бәрін тікелей сұрастыру керек. Бұл менеджердің жүктемесін арттырып, сатылымды баяулатады.
Үшіншіден, тапсырыстардың орындалу барысын бақылау жүйесі жоқ. Бір тапсырыс дайын болды ма, жоқ па - тек адам жадында сақталады. Статус өзгерген кезде клиентке хабарлаудың да жүйелі механизмі жоқ.
Төртіншіден, қаржылық аналитика толығымен қолмен жасалады. Ай соңында менеджер тапсырыс дәптерлерін, хабарлама тарихтарын жинап, Excel-ге өзі теруге мәжбүр болады. Бұл уақыт шығыны да, қателік қаупі де жоғары.
Осы талдаудан туындаған жобаның негізгі мақсаттары:
- Тауарларды (ағаш есіктер, үстелдер, жиһаздар) электрондық каталог түрінде ұсыну;
- Тапсырысты веб-арқылы қабылдап, дерекқорда сенімді сақтау;
- Тапсырыс статусын басқару және клиентке хабарлау механизмін іске асыру;
- Кәсіпорынның қаржылық көрсеткіштерін автоматты есептеп, Excel есебі ретінде экспорттау;
- Жүйені жоғары жүктемелерге төтеп бере алатын микросервистік архитектурада жобалау.
Жобаның негізгі мақсаты - «Stolyarniy dvor» цехы үшін екі бағытта жұмыс істейтін веб-қосымша жасау: тұтынушы жағынан тауар таңдап тапсырыс беруге, ал басшылық жағынан барлығын бір орталықтан бақылауға мүмкіндік беру.
Бірінші міндет - архитектура. API шлюзін (api-gateway) арқылы байланысқан екі тәуелсіз микросервис: products-service (тауарлар) және orders-service (тапсырыстар). Бұл жүйенің тұрақтылығын қамтамасыз етеді [3].
Екінші міндет - интерфейс. Клиентке арналған дүкен бөлігі мен басқарушыға арналған аналитика панелі бір жүйеде біріктірілді. Графиктер мен визуализация React арқылы іске асырылды.
Үшінші міндет - бизнес-логика. Күрделі уақыт фильтрациясы, алдыңғы кезеңмен салыстыра пайыздық өсімді есептеу, Excel форматында есеп жүктеу.
Төртінші міндет - интеграция. PostgreSQL дерекқоры, Telegram Bot арқылы лезде хабарлама, Git/GitHub арқылы нұсқа бақылауы, Docker контейнерлері мен Render платформасына орналастыру.

1.3 Жүйеге қойылатын функционалдық және техникалық талаптар

Функционалдық талаптар-бұл ақпараттық жүйенің пайдаланушылар мен бизнеске қатысты қандай нақты тапсырмаларды, операциялар мен функциялардың орындалуын қамтамасыз етуге міндетті екенін декларативті түрде анық сипаттайтын ережелер мен шарттар жиынтығы. Мұндай талаптар жоба техникалық тапсырмасының негізгі бөлігін құрайды және барлық жобалау шешімдерінің бастапқы нүктесі болып табылады [5].
Клиенттік (ашық) функционал:
-Пайдаланушы тіркелмей-ақ өнімдердің толық электрондық каталогын еркін шола алуы, әр тауардың сипаттамасын, материалын, өлшемдерін және нарықтық бағасын егжей-тегжейлі оқи алуы тиіс; каталог суреттерімен, атаулармен және категориялармен толықтырылуы керек;
-Тұтынушы жеке тапсырысты (CustomOrder, яғни өзіндік пішін мен өлшемде жасалатын бұйым) немесе каталогтан таңдалған дайын стандартты өнімдерді себет (cart) механизмі арқылы бірнеше тауар бір мезетте жинақтап, аты-жөнін, байланыс телефон нөмірін, жеткізу мекен-жайын толтырып, тапсырысты толыққанды ресімдей алуы тиіс; форманың барлық өрістері серверге жіберілмей тұрып клиент жағында да валидациядан өтуі шарт;
-Тапсырыс сәтті жіберілгеннен кейін клиент браузер экранында жасыл белгімен расталған хабарлама (Success Notification) көріп, тапсырыстың дерекқорға сәтті тіркелгендігі туралы нақты визуалдық растау алуы; тапсырыстың бірегей идентификатор нөмірі жүйеде сақталуы керек.
Басқару (әкімшілік) функционал:
-Жүйенің авторизациялық модулі арқылы тек белгіленген рөлдегі (ADMIN) пайдаланушылар ғана қауіпсіз логин/пароль механизмімен жүйеге кіріп, барлық ағымдағы, өңделіп жатқан (PROCESSING), аяқталған (DONE) және жаңа (NEW) тапсырыстардың толық тізімін бір орталықтандырылған басқару тақтасынан (dashboard) нақты уақыт режимінде бақылап, қарай алуы керек;
-Әкімші тапсырыстың өмірлік циклі бойынша статустарын (мысалы: ЖАҢА-ӨНДІРІСТЕ-ДАЙЫН-ТАБЫСТАЛДЫ) икемді өзгерте алуы, клиент деректерін түзете алуы және қажет болған жағдайда тапсырысты жүйеден жоя алуы тиіс; ол үшін интерактивті кесте (Data Grid) мен редактирлеу формасы болуы керек;
-Өнімдер микросервисіндегі (products-service) деректерді, яғни тауарлардың атауын, сипаттамасын, бағасын, суреттерін және категориясын толыққанды CRUD (Create-жасау, Read-оқу, Update-жаңарту, Delete-жою) операциялары арқылы тікелей интерфейстен икемді басқара алу мүмкіндігі қамтамасыз етілуі тиіс;
-Аналитикалық тақтада (Analytics Dashboard) әкімші таңдаған уақыт кезеңі (күн, апта, ай, тоқсан) ішіндегі жаңа тапсырыстардың жалпы санын, барлық тапсырыстардың жиынтық сомасын, кәсіпорынға түскен жалпы қаражатты (выручка) және орта есеппен бір тапсырыстың құнын (орташа чек) автоматты есептеп көрсетуі; осы сандық мәліметтерді динамикалық сызықтық екі графикте (Dual-Line Chart) уақыт шкаласымен бірге визуализациялауы; алдыңғы кезеңмен салыстырғандағы өсім немесе төмендеу пайызын жасыл/қызыл индикаторлармен бейнелеуі; аналитиканы кез-келген уақыт аралығымен сүзгілеуге (фильтрлеуге) толық мүмкіндік берілуі тиіс;
-Барлық аналитикалық көрсеткіштер жиынтығын, тапсырыстар тізімін және клиент деректерін бір рет батырма басу арқылы корпоративтік Excel (.xlsx) файлына автоматты жүктеп алу (экспорттау) функциясы іске асырылуы керек; файл оқуға ыңғайлы форматта (тақырып бағандары қалың, деректер кестесімен) жасалуы тиіс.
Жүйелік функционал:
-Жаңа тапсырыс веб-жүйеге сәтті тіркелген сәтте сервер (Orders Service микросервисінің ішінде) Telegram Bot API арқылы цех менеджеріне немесе өндіріс тобына арналған белгіленген топтық чатқа не жеке парақшаға тапсырыс деректерін (тұтынушы аты, байланысы, тауарлар тізімі, жалпы сома) қамтыған толыққанды хабарламаны автоматты және лезде жеткізуге міндетті; бұл процесс негізгі тапсырыс жасау транзакциясын тоқтатпай, асинхронды режимде орындалуы шарт.

 

Сурет 1. Жүйенің Use-Case (Қолдану нұсқалары) диаграммасы.

Архитектура: бағдарлама бір монолитті серверде емес, HTTP-хаттамасы арқылы өзара байланысқан API Gateway (шлюз) және тәуелсіз микросервистер базасында жұмыс істеуі шарт.
Қауіпсіздік: серверлік бөлім мен фронтенд арасындағы сұраныстар CORS (Cross-Origin Resource Sharing) саясаттары арқылы қорғалуы тиіс [4]. Базалық авторизация параметрлері мен қауіпсіздік кілттері (Token) .env қоршаған орта айнымалыларында сақталуы қажет.
Қолжетімділік: жүйе және оның API интерфейстері әуелі локальді түрде Docker арқылы сыналуы керек, кейіннен Render бұлтты платформасында тұрақты жұмыс істеуі керек.
Тиімділік: аналитикалық деректерді қамтитын үлкен мәліметтер жиынтығына сұраныс жасаған кезде API жауап беру уақыты микросервистің архитектурасы мен дұрыс жазылған Prisma сұраныстарының арқасында оңтайлы болуы керек.
Бақылау мен Тарих: барлық енгізілген кодтық өзгерістер міндетті түрде Git арқылы бақыланып, үнемі GitHub репозиторийіне синхрондалып отыруы қажет. Бұл командалық жұмысқа немесе кейін масштабтауға негіз жасайды.

2 ҚОСЫМШАНЫҢ ИНТЕРФЕЙСІ МЕН ФУНКЦИОНАЛДЫҒЫН ЖОБАЛАУ


2.1 Жүйенің архитектурасын таңдау және негіздеу

«Stolyarniy dvor» жобасын ең басынан бастап жобалау кезінде, бағдарламаның бүкіл өмірлік циклін (SDLC) айқындайтын ең маңызды стартегиялық қадам-дұрыс архитектуралық үлгіні (паттернді) таңдау болды. Қазіргі жоғары жүктемелі (high-load) веб-әзірлеу тәжірибесінде екі негізгі тәсіл қарсы қойылады: дәстүрлі монолитті архитектура (Monolithic Architecture) және заманауи микросервистік архитектура (Microservices Architecture) [4]. Өнімнің архитекторы ретінде нарықтық аналитикалық талдау және жобаның болашақтағы кем дегенде 5 жылдық даму перспективаларын, жүктемелерді бағалау барысында, бастапқы құрылымы күрделі болғанына қарамастан, микросервистік тәсіл пайдасына біржақты шешім қабылданды. Жүйенің іргетасы ретінде бұл таңдау оның функционалдығын ауыртпалықсыз кеңейтуде, жаңа мүмкіндіктерді жылдам енгізуде өте маңызды рөл атқарады.
Неліктен дәстүрлі Монолитке (Monolith) таңдау түспеді?
Монолитті архитектура-бұл бағдарламалық қамтамасыз етуді әзірлеудің ең көне және қарапайым тәсілі. Онда клиенттік сұраныстарды өңдеу, деректер қорымен (Database) жұмыс істеу, қолданушыларды тіркеу, есептер мен аналитика генерациялау, және Telegram-ға хабарлама жіберу сияқты барлық бизнес-логика бір ғана үлкен серверлік қосымшаның (мысалы, жалғыз Node.js немесе Spring Boot сервері) ішіне тығыз біріктіріледі (tightly coupled). Монолитті жүйенің бастапқыда үлкен артықшылығы бар: оны құру (development), тестілеу (testing) және кез-келген хостингке бір файл ретінде орналастыру (deployment) өте жеңіл. Алайда, оның бірқатар елеулі әрі қауіпті кемшіліктері бар, олар жобаның кодтық базасы өскен сайын айтарлықтай инженерлік кедергіге айналады:
- Жүктемені арттыру мүмкіндігінің (Scalability) төмен тиімділігі: Егер жүйеге мереке қарсаңында жаңа тапсырыстар легі көптеп түсіп, серверге кенеттен күш түссе, монолитті жүйеде жүктелген «тапсырыс қабылдау модулін» ғана жеке кеңейту мүмкін емес. Бұл жағдайда кітапханалармен, ауыр аналитика функцияларымен бірге тұтас монолитті бағдарламаны толық көшірмелеу (scale out) қажет болады. Бұл сервер ресурстарын (CPU, RAM) тиімсіз жұмсауға және хостинг төлемдерінің негізсіз өсуіне әкеледі.
- Қателіктерге төзбеушіліктің (Fault Tolerance) төмендігі (Бірыңғай істен шығу нүктесі - Single Point of Failure): Монолит қолданбаның ішіндегі кез-келген шағын қателік бүтіндей жүйенің құлауына себеп болуы мүмкін. Мәселен, өнімдер каталогындағы (products) суретті жүктеу кезіндегі кішкене логикалық қателік немесе жадтың ағып кетуі (Memory Leak) бүкіл сервердің өшуіне, оның ішінде өте маңызды қаржылық тапсырыс қабылдау (checkout) жүйесінің тоқтап қалуына әкеп соқтырады. Бұл бизнес үшін тікелей қаржылық шығын (downtime loss).
- Технологиялық тәуелділік пен икемсіздік (Technology Lock-in): Барлық жүйе бір ғана бағдарламалау тіліне немесе бір ғана фреймворкқа өмір бойы байлаулы болады. Болашақта аналитика үшін өте тиімді саналатын Python (Pandas/NumPy) модулін қосқымыз келсе, монолит бұған жол бермейді немесе архитектураны басынан жазуға мәжбүрлейді. Сонымен қатар, код базасы үлкейген сайын, жаңадан келген әзірлеушілерге жүйенің кодын оқып-түсіну (onboarding process) апталарға созылуы мүмкін.
Заманауи Микросервистік архитектураның шешімі:
Дәл осы тәуекелдерді болдырмау үшін жобада Доменге бағдарланған дизайн (DDD - Domain Driven Design) принциптері қолданылды. Бизнес-логика мағынасына қарай бірнеше оқшауланған контексттерге (Bounded Contexts) бөлінді. Нәтижесінде, API Gateway (HTTP-шлюз маршрутизаторы) және тәуелсіз екі басты бэкенд-сервис тораптарына (nodes) негізделген микросервистік архитектура (microservices architecture) таңдалды. Бұл архитектураның негізгі әрі функционалдық изоляциясы бар компоненттері мыналар:
1. API Gateway (Сұраныстар шлюзі)-бұл ғаламтордан келетін барлық клиенттік (фронтенд немесе мобильді қосымша) сұраныстар тікелей келіп түсетін жүйенің жалғыз қоғамдық және орталықтандырылған кіру нүктесі (Single Entry Point, 8080 портында жұмыс істейді). Бұл сервис өзі ешқандай бизнес-логиканы немесе базалық операцияны (CRUD) орындамайды. Оның басты мақсаты-«Reverse Proxy» қызметін атқару. Ол сұраныс адресіне (URL path) қарай ішкі трафикті тиісті жабық микросервистерге бағыттайды (Routing). Мысалы, `/api/products` адресіне келген пайдаланушы сұранысын ол `Products Service`-ке, ал `/api/orders` адресін `Orders Service`-ке мұқият бағыттайды. Сонымен қатар, Gateway қауіпсіздік қалқаны ретінде жұмыс істейді: барлық рұқсатсыз шабуылдар, спам сұраныстар осы жерде бұғатталады (Rate Limiting), ал микросервистер интернеттен тікелей шабуылға ұшыраудан (DDoS) қауіпсіз ішкі желіде жасырылады.
2. Products Service (Өнімдер қызметі, порт 3001)-тек қана өнімдер каталогын, олардың сипаттамаларын, материал параметрлерін, бағаларын және жоғары сапалы сурет сілтемелерін сақтап, клиентке өте жылдам мәлімет қайтаратын жеке, тәуелсіз шағын қызмет. Ол цех каталогын басқару үшін жүйенің басқа бөліктеріне қарамастан автономды жұмыс істейді (Stateless nature). Бұл сервистің ең басты ерекшелігі-оның оқуға (Read-heavy) бағытталғандығы, сондықтан оны болашақта кэштеу (Redis) арқылы мыңдаған рет еш кедергісіз масштабтауға болады. Мұнда ешқандай ауыр логика жүрмейді, тек клиентке ақпаратты тез көрсету мақсаты көзделеді.
3. Orders Service (Тапсырыстар қызметі, порт 3002)-бұл бүкіл интернет-дүкеннің «миы», яғни транзакциялар мен күрделі іскерлік операцияларды орындайтын негізгі бизнес-логика қызметі (Write-heavy) [8]. Мұнда тұтынушылардың жеке деректерін тіркеу, себеттен келген күрделі тапсырыстар мен бірнеше тауарларды өңдеу (Checkout process), төлемдердің сомасын есептеу, админ-панельге арнайы ауаны тазартатын аналитикалық есептеулер (Data Aggregation) жүргізу сияқты ең ауыр операциялар орындалады. Сондай-ақ, өндіріс админдеріне Telegram арқылы лезде хабарлама жіберу функциясы мен пайдаланушылардың Аутентификация / Авторизация (JWT токендер, bcrypt пароль хэштері) механизмдері де осы қауіпсіз оқшауланған ортада шоғырланған.
Бұл архитектуралық шешім жоғарыда аталған сүреңсіз монолиттің барлық кемшіліктерін түбегейлі жояды (Mitigation of Risks). Инженерия заңдылығы бойынша ең басты артықшылық мынада: егер Тапсырыстар қызметі (Orders Service) күрделі есептеулер жүргізіп жатып немесе дерекқор қатесіне (Database timeout) ұрынып істен шыққан (crashed) күннің өзінде, Өнімдер (Products Service) қызметі еш кедергісіз өз жұмысын жалғастыра береді. Яғни, сайтқа кірген тұтынушылар үшін дүкен өшіп қалмайды, олар каталогты ашып, есіктер мен үстелдердің суреттерін қарай береді (Graceful Degradation). Тек тапсырыс беру түймесі уақытша істемей тұруы мүмкін. Бұл заманауи жоғары технологиялық (Enterprise) компаниялар қолданатын ең сенімді тәсіл болып саналады.

 

Сурет 3. Жобаның микросервистік архитектура схемасы.

Жобаның тұрақтылығы мен сүйемелдену мүмкіндігі (maintainability) тікелей дұрыс таңдалған технологиялар жиынтығына байланысты. Тәменде технологиялардың балама нұсқаларымен салыстырмалы талдауы келтірілген.
Фронтенд: React.js (Vite) vs Vue.js / Angular. Интерфейс жасау үшін React 18 кітапханасы таңдалды. Angular фреймворкі оқу қисығы (learning curve) ауыр, ол үлкен корпоративтік жобалар үшін жақсы (enterprise). Vue.js жеңіл болғанымен, React нарықта абсолютті көшбасшы болып табылады, оның қауымдастығы (community) және дайын құралдар жиынтығы өте бай. Жобада графиктер (Recharts) мен аналитикалық дашборд жасау қажет болғандықтан, React компоненттік тәсілі (Component-Based Approach) ең қолайлы таңдау болды [7]. Webpack орнына Vite қолданылды-ол HMR протоколы арқылы әзірлеу процесін әлдеқайда жылдамдатады [16].
Бэкенд: Node.js (Express) vs Python (Django) / Java (Spring Boot). Серверлік логика үшін Node.js платформасындағы Express.js таңдалды. Java Spring Boot шағын микросервистер үшін конфигурациялауы тым ауыр. Python Django да ыңғайлы болғанымен, Node.js асинхронды I/O моделімен ерекшеленеді-бір мезетте мыңдаған сұранысты аз жад жұмсай өңдей алады. Express.js-тің минимализмі микросервис форматына дәл сәйкес келді [10]. Фронтенд пен бэкенд бір тілде (JavaScript) жазылуы-жалпы код парадигмасын оңайлатты [4].
Деректер қоры: PostgreSQL vs MongoDB. Деректер қоры ретінде реляциялық PostgreSQL 16 таңдалды. Электрондық дүкен мен тапсырыстар жүйесі үшін қатаң структурасы жоқ MongoDB сияқты NoSQL шешімдерін қолдану қате болып табылады, себебі қаржылық транзакциялар, пайдаланушы мен оған тиесілі тапсырыстар, тапсырыстағы жекелеген тауарлар қатаң (ACID) реляциялық байланыста болуы міндетті [12]. Тек PostgreSQL ғана үлкен көлемдегі аналитикалық сұраныстарды жылдам әрі дәл өңдеуге қажетті математикалық агрегация функцияларына және GIN индекстеріне ие.
ORM (Нысан-реляциялық бейнелеу): Prisma vs Sequelize. Таза (Raw) SQL сұраныстарын жазу қателіктерге толы болуы мүмкін әрі бағдарлама кодын оқуды қиындатады. Sequelize сияқты ескі ORM-дар типі қатаң тексерілмеген (Not TypeSafe) сұраныстар жасайды. Жоба үшін Prisma ORM технологиясы таңдалды. Ол схеманы (schema.prisma) нақты декларативті түрде сипаттап қана қоймай, объектілі-бағытталған ортада (JavaScript/TypeScript) деректер қорының қатерін автоматты түрде автотолтырумен (auto-completion) қамтамасыз етеді және миграцияларды (prisma migrate) өздігінен жасайды [13].
Стильдеу: Tailwind CSS. Классикалық каскадты стиль кестелерінің (Plain CSS) немесе SASS/LESS препроцессорларының орнына утилитарлы Tailwind CSS фреймворкі таңдалды. Ол HTML-компоненттері ішінде CSS-кластарды логикалық қолдану арқылы қосымша файл көлемін барынша азайтады (purge mechanism), ал стилистикалық көріністі ең жоғары деңгейде қалдырады.

2.2 Жүйенің құрылымын және деректер қорының моделін жобалау

«Stolyarniy dvor» жобасының микросервистік архитектурасының қатаң талаптарына сәйкес, деректер қоры монолитті бірыңғай жүйе ретінде емес, микросервистер қағидасы бойынша «әр қызметке жеке дерекқор» (Database per Service) үлгісінде екі тәуелсіз сегментке немесе кестелер пулына бөлінген (schema isolation). Негізгі қозғалтқыш (engine) ретінде транзакциялық сенімділігі өте жоғары PostgreSQL реляциялық дерекқоры таңдалды. Ол келесідей негізгі объектілер (Кестелер - Tables), қатынастар (Relations) мен мәндерге негізделе отырып жобаланды:
1. Өнімдер қызметінің (Products Service) схемасы:
Бұл қызмет өзінің оқшауланған тәуелсіз құрылымын пайдаланады. Мұндағы басты мақсат-каталогты басқару.
- `Product` (Тауар) кестесі: Цехтың барлық шығаратын ағаш өнімдерінің (есіктер, үстелдер, шкафтар) электрондық каталогы қызметін толыққанды атқарады. Оның ішкі атрибуттары: `id` (PK, Primary Key-тиісті тауардың қайталанбас идентификаторы, autoincrement форматында өседі), `name` (өнімнің коммерциялық атауы, String форматында), `description` (өнімнің материалдары мен сапасын көрсететін толық сипаттамасы, Text), `price` (базалық сатылым бағасы, теңгемен, Float), `images` (клиентке көрсетілетін веб-суреттер массивтері, Array of Strings), және жүйенің ішкі аудиті үшін қажетті `createdAt` (құрылған уақыты), `updatedAt` (жаңартылған уақыты) сияқты уақыттық белгілер (timestamps). 
2. Тапсырыстар қызметінің (Orders Service) схемасы:
Бұл бағдарламаның жүрегі саналатын сервис. Мұнда пайдаланушы мен тапсырысты, сондай-ақ қаржылық операцияларды байланыстыратын аса маңызды бизнес кестелер сақталған. Кестелер өзара Foreign Key (шеткі кілттер) арқылы дәл қатынаста орнатылған (Referential Integrity):
- `User` кестесі: Жүйедегі әкімшілік және клиенттік басқару үшін қажетті басты кесте. Оның атрибуттары: `id` (PK, бұл жолы қауіпсіздік үшін UUID форматы таңдалған), `name` (қолданушының аты-жөні), `phone` (жүйедегі бірегей мән немесе unique field, ол авторизация кезінде логин ретінде де қолданылады). Жүйе қауіпсіздігі үшін пайдаланушының құпия сөзі (password) базада ашық мәтінмен сақталмайды, ол заманауи `bcrypt` криптографиялық алгоритмі арқылы күрделі хэш (hash) түрінде сақталады. Сондай-ақ, кестеде `role` өрісі бар (ADMIN немесе USER екенін білдіретін ENUM типіндегі мән), бұл рөлдерге негізделген қатынасты басқару (RBAC) үшін қолданылады. Тұтынушының бүкіл тарихын бір жерден көру үшін бұл кесте 1-көпке (1-to-N) қатынасымен `Order` кестесіне тікелей сілтеме жасайды.
- `Order` кестесі: Жүйенің ең негізгі транзакциялық және қаржылық кестесі. Мұнда тұтынушының аты-жөнінен бөлек, логистикаға арналған `address` (жеткізу мекенжайы), `paymentMethod` (төлем тәсілі: банк картасы немесе қолма-қол), `totalPrice` (алгоритмдер арқылы есептелген жалпы сома), `status` (тапсырыстың өмірлік циклін білдіретін күйлер: NEW - жаңа, PROCESSING - өңделуде, READY - дайын, DONE - клиентке табысталды) сияқты өте маңызды бизнес-көрсеткіштер сақталады. Бұдан бөлек, цехтың спецификалық ерекшеліктеріне байланысты архитектура әдеттегі (PRODUCT) каталогтық тапсырыстардан бөлек, клиенттің жеке сызбалары немесе кастомды тапсырыстарын (CUSTOM) қабылдай алуы үшін `type` Enum-өрісі қосылған. Бұл жағдайда кастомды өнімге арнайы `customTitle`, `customDescription` сипаттама атрибуттары қосылады.
- `OrderItem` кестесі: Сауда логикасында бір тапсырыс (Order) ішінде бірнеше түрлі тауар (мысалы, 1 үстел және 4 орындық) қатар жүруі әбден мүмкін болғандықтан орнатылған қатынас кестесі (N-to-N аралық көпір кестесі). Мұнда `id` (PK), `orderId` (Orders кестесіне қатаң физикалық сілтеме), `productId` (Products сервисіндегі өнімнің логикалық/виртуалды ID-сі), `quantity` (тиісті тауардың нақты саны), және ең бастысы тапсырыс жасалған сәттегі `price` (фиксацияланған сатылым бағасы) атрибуттары сақталады. Бағаның фиксациялануы өте маңызды, өйткені тауардың болашақтағы базалық бағасы инфляцияға байланысты өзгеруі мүмкін, бірақ ескі тапсырыстар тарихында клиенттің бұрын қаншаға сатып алғаны өзгеріссіз қалуы тиіс.
Сервистер аралық байланыс және деректер біріктіру ерекшеліктері (API Composition & Hydration):
Микросервистік архитектураның басты қиындығы-деректер қорының оқшаулануы (Bounded Context). Яғни, `OrderItem` кестесіндегі `productId` қатаң SQL-шеткі кілті (Foreign Key Constraint) ретінде пайдаланылмайды, себебі өнімдер кестесі физикалық түрде басқа (Products) базада жатыр. Ол жай ғана бүтін сан (Integer) мәні ретінде сақталады. Ендеше, клиент өзінің тапсырыстар тарихында тауардың құр ID-сін емес, толық суретін және атын көруі үшін арнайы механизм жасалған. 
Тапсырыстар тізімін оқу (fetch) кезінде Orders Service өз ішінде HTTP/REST протоколы арқылы Products Service-ке арнайы асинхронды сұраныс жіберіп, сол тапсырыс ішіндегі барлық сәйкес келетін өнімдердің суреті мен атауларын алып келеді. Осы екі жақтан келген мәліметтерді Node.js жадында біріктіріп (hydration әдісі немесе API Composition паттерні), фронтендке толыққанды байытылған объект ретінде қайтарады. Бұл тәсіл микросервистер арасындағы кросс-дерекқорлық SQL JOIN-дардан (өте баяу процедура) бас тартуға мүмкіндік береді. Ол деректердің домендік құпиялығын (data integrity) және транзакциялық шекараларды (transaction boundaries) сақтай отырып, фротендке монолитті архитектураға тән қолайлылықты (бір ғана API шақыруын) ұсынатын озық бағдарламалық инженерлік үлгі болып табылады [23].

 

Сурет 4. Деректер базасының ER-моделі (Кестелер қатынасы).

Клиенттік веб-интерфейс бэкендпен қатаң түрде RESTful стандарты арқылы қарым-қатынас жасайды [11]. Орталықта тұрған API Gateway 8080 портында жұмыс істеп, барлық HTTP-сұраныстарды (/api/…) қабылдап, http-proxy-middleware арқылы төмендегідей ішкі бағыттауды жүзеге асырады:
-маршрут: /api/products/* -> proxy -> http://products-service:3001
-маршрут: /api/orders/* -> proxy -> http://orders-service:3002
-маршрут: /api/auth/* -> proxy -> http://orders-service:3002
-маршрут: /api/users/* -> proxy -> http://orders-service:3002
Нәтижесінде, клиент үшін барлық қызметтер бір серверден (8080) келіп тұрғандай иллюзия (API фасады) жасалынады. Бұл CORS (Cross-Origin Resource Sharing) проблемаларын болдырмайды.
Orders-service ішіндегі негізгі API эндпоинттері:
-POST /orders: жаңа тапсырысты (стандартты немесе custom) қабылдау. Егер файлдармен келсе, multer кітапханасы мульти-парталы (multipart/form-data) суреттерді uploads каталогына сақтайды.
-GET /orders: бақылау тақтасы үшін барлық тапсырыстар тізімін сұрыптап (createdAt бойынша desc) қайтару. verifyToken және requireAdmin қауіпсіздік мидлварларымен (middleware) қорғалған.
-PUT /orders/:id/status: тапсырыс статусын ауыстыру (Жаңа-Орындалуда-Дайын).
Auth-service логикасы: POST /auth/register және POST /auth/login арқылы bcrypt арқылы телефон/пароль байланысын тіркеп, жауап ретінде құрамында қолданушының ID мен рөлі (role) бар жасырын JWT (JSON Web Token) қайтарады [19]. Келесі кез-келген жеке (private) сұраныстар кезінде веб-қосымша осы токенді HTTP-Header (Authorization: Bearer <token>) арқылы жіберіп отырады. Сервер оны верификациялап (verify), рұқсат деңгейіне қарай мағлұмат береді.

 

Сурет 5. API Gateway маршрутизация логикасының блок-схемасы.

2.3 Жүйе компоненттерінің өзара әрекеттесуін ұйымдастыру және іске асыру

Әкімшілік панельдің ішінде (Admin Dashboard) тек қана мәліметтерді қосу немесе жою секілді кәдімгі CRUD (Create, Read, Update, Delete) операциялары ғана емес, сонымен қатар «Stolyarniy dvor» кәсіпорнының басшылығына арналған кешенді бизнес-аналитика жүйесі (BI - Business Intelligence) құрастырылды. Кез-келген бизнестің табыстылығын қадағалау үшін қаржылық және статистикалық есептер өте маңызды рөл атқарады. Тапсырыс берушілердің мінез-құлқын, кірістер ағынын және сату динамикасын түсінуге бағытталған есептер логикасы келесідей құрылымдалған алгоритмдермен жүзеге асырылды:
1. Кешенді Дерекқор сұраныстары және Агрегация (Data Aggregation & Prisma Logic): Есептеу процестері клиенттік браузерде (фронтендте) жасалмайды, керісінше барлық математикалық жүктеме бэкенд серверіне (Node.js негізіндегі Orders Service) аударылған. Бэкенд деректер базасынан ақпарат алу үшін қауіпсіз әрі өте жылдам `Prisma ORM` жүйесін қолданады. Админ панелінде әкімші белгілі бір уақыт аралығын (Мысалы: «Осы апта», «Осы ай» немесе «Өткен 30 күн») құламалы тізімнен (Dropdown) таңдаған кезде, фронтенд серверге ISO-8601 форматындағы бастапқы және аяқталу күндерін (Date Range) жібереді. Серверде `Prisma.$transaction` және мәліметтерді топтастыру (`groupBy`), сондай-ақ қосу функциялары (`_sum`, `_count`) құралдары арқылы көрсетілген уақыт ішіндегі барлық аяқталған тапсырыстар жинақталады. 
Сол жиналған ауқымды массивтен жалпы табыс (Total Revenue) және орташа чек (Average Check) математикалық модельдермен қысқа уақыт аралығында есептеледі. Пайыздық өсімді (динамиканы - Growth percentage) табу үшін жүйе автоматты түрде осы таңдалған уақыт ауқымымен бірдей көлемдегі тарихи (өткен) кезең деректерін базадан қосымша сұратып, екі кезеңнің көрсеткіштерін салыстырады. Қорытындылай келе, табыстың мысалы `+15.4%` өсуін немесе `-5.2%` төмендеуін жасыл және қызыл түстермен дәлдікпен анықтап, фронтендке жібереді.
2. Графикалық визуализация және Аналитика (Recharts Library): Адам миына құрғақ сандардан гөрі визуалды бейнелерді қабылдау әлдеқайда оңай. Сондықтан React клиентінде әлемге әйгілі `Recharts` React-компонент кітапханасы толықтай интеграцияланды. Деректер қорынан өңделіп келген JSON форматындағы массивтер (Күн-Табыс сомасы-Тапсырыс саны) Екі-сызықты динамикалық графикаға (Dual-Line Chart) бейнеленеді. Графиктің X (көлденең) өсінде күндер немесе апталар уақыт шкаласымен орналасады. Ал графиканың оң және сол жақ шеттерінде, яғни екі Y (тігінен) өсінде-бір мезгілде қаржылық табыс (теңгемен) және сатылған тауарлар саны шкаласы көрсетіледі. Сонымен қатар, графика ResponsiveContainer құралымен орамдалғандықтан, ол монитордың немесе ұялы телефонның экранына сай автоматты түрде өз пішіні мен өлшемін өзгертеді. Тінтуірмен (mouse) графиктің кез келген нүктесіне жақындатқан кезде, әдемі Tooltip (көмекші терезе) шығып, сол күнгі нақты табыс көлемін интеракивті түрде көрсетеді. Бұл цех иесіне кірістер ағынының трендін (маусымдық сұраныс, жарнаманың тиімділігі) визуалды бағалауға және нақты фактілерге сүйенген маркетингтік шешімдер қабылдауға мүмкіндік береді [25].
3. Корпоративтік Excel генераторы (Automated Report Export): Мекеменің күнделікті жұмысында сандық графиктерден бөлек, қағазға басып шығаруға немесе бухгалтерлік 1С жүйелеріне импорттауға арналған ресми құжаттар өте маңызды. Осы қажеттілікті өтеу үшін және әрбір есепті корпоративтік бухгалтерлік ревизияға немесе мұрағаттауға (архивтеу) дайындау мақсатында Node.js ортасында `exceljs` қуатты кітапханасы пайдаланылды [24]. 
Бұл шешімнің ең басты архитектуралық ерекшелігі-ресурсты көп қажет ететін файлды қалыптастыру процесі тұтынушының әлсіз браузерінде емес, тікелей серверде (бэкендте) жүзеге асады. Сервер барлық тапсырыс мәліметтерін айналдырып, кесте бағандарының өлшемдерін (column widths), қаріп түрлерін (font styling - қалың әріппен жазылатын тақырыпшалар) және ұяшық түстерін форматтайды. Содан кейін сервер Node.js Buffer (жад ағыны) арқылы лезде `.xlsx` ұлғайтылған файлын қалыптастырып, оны Blob (Binary Large Object) объектісі ретінде жүктеп алу эндпоинтын ұсынады. Дайын шыққан Excel файлы өте түсінікті форматта (ID нөмірі, Тауарлар тізімі, Тапсырыс берушінің деректері, Баға, және Статус) ұсынылады. Бұл тәсіл, әсіресе 1000-нан астам тапсырысы бар файлдарды экспорттау кезінде, сервердің жадын үнемдеп, есептің сенімді әрі қатесіз жүктелуіне 100% кепілдік береді.

 

Сурет 6. Аналитика мен Excel есеп қалыптастыру алгоритмі.

Клиенттік веб-интерфейстің архитектурасы пайдаланушы назарын негізгі мақсатқа – тауарды таңдауға және тапсырыс беруге шоғырландыру үшін ықшамдалған UI/UX шешімдеріне негізделді. Бағдарлама интерфейсі React компоненттік құрылымы арқылы жасалды. Парақтар арасындағы навигация Single Page Application (SPA) қағидасына сәйкес `React Router v6` маршрутизаторы арқылы жүзеге асады, бұл DOM манипуляциялары арқылы бетті жаңартпай динамикалық өтулерді қамтамасыз етеді. Интерфейстің негізгі құрылымдық модульдері келесідей жобаланды:
- Клиенттік модуль (`/`, `/products`, `/contacts`, `/cart`): Ашық дүкен фасады. Басты бет (`/`) өндіріс артықшылықтары мен негізгі тауарларды көрсететін ақпараттық құрылымға ие. Тауарлар каталогы (`/products`) CSS Grid және Flexbox ережелеріне негізделіп, экран өлшеміне бейімделетін (responsive) орналасуды қамтамасыз етеді. Себет (`/cart`) модулінде таңдалған тауарлар мен жалпы сома көрсеткіштері Zustand күй менеджері арқылы нақты уақыт режимінде жаңартылады.
- Аутентификация модулі (`/login`, `/register`): Пайдаланушыларды тіркеу және жүйеге кіру формалары екіге бөлінген экран (Split-screen) тәсілімен орналастырылды. Экранның бір бөлігінде ақпараттық визуалдау, ал екінші бөлігінде форма енгізу өрістері (input fields) орналасқан. Дұрыс емес мәлімет енгізілген жағдайда, жүйе пайдаланушы сценарийлерін ескере отырып, форма валидациясының қателерін мәтіндік нұсқаулармен қайтарады.
- Пайдаланушы профилі (`/profile`): Тұтынушының жүзеге асырған тапсырыстарының тарихы мен статус өзгерістерін («Жаңа», «Өңделуде», «Аяқталды») бақылауға арналған жабық кеңістік. Бұл модуль тұтынушы мен кәсіпорын арасындағы өзара әрекеттесуді жүйелеп, тапсырыс тарихы туралы ақпараттың қолжетімділігін ұйымдастырады.
- Басқару тақтасы (`/admin`): Дерекқорда `ADMIN` рөлі бекітілген тұлғаларға арналған басқару интерфейсі. Құрылымдық жағынан бүйірлік навигациялық мәзірден (Sidebar), орталық аналитикалық графикадан (Recharts) және тапсырыстарды басқару кестесінен (Data Grid) тұрады. Бұл компонент кәсіпорынның статистикалық мәліметтерін өңдеп, шешім қабылдау процестерін қолдауға бағытталған.
Қазіргі заманғы электрондық коммерция саласында тапсырыстарды өңдеудің жылдамдығы (SLA - Service Level Agreement) бизнестің бәсекелестік қабілетін тікелей анықтайды. Тұтынушы өзінің тапсырысын жіберген сәтте, оны цех менеджері немесе сату бөлімі бірден көріп, клиентке қайта хабарласуы өте маңызды. Осы процесті, яғни тапсырыс берушіні күттіріп алмау үшін және цех әкімшілігін жедел әрі дәл ақпаратпен қамтамасыз ету мақсатында жүйеге лездік хабарландыру (Instant Push Notification) модулі енгізілді [14]. Бұл шешім ретінде дәстүрлі SMS немесе Email жіберуден гөрі әлдеқайда жылдам әрі ыңғайлы Telegram Bot API экожүйесі таңдалды. Оның бағдарламалық алгоритмі архитектуралық тұрғыда келесідей құрылымдалған сатылармен жүреді:
1) Алғашқы транзакциялық саты: Бэкендте (Orders Service микросервисінде) клиенттен келген тапсырыс мәліметтері алдымен Zod кітапханасы арқылы қатаң валидациядан өтеді. Барлық форматтардың дұрыстығы расталғаннан кейін, тапсырыс PostgreSQL реляциялық базасына сенімді түрде сақталады.
2) Хабарламаны генерациялау (String Formatting & Templating): Дерекқор тапсырыстың сәтті сақталғандығы туралы жауап қайтарған бетте асинхронды қосалқы бағдарлама (background task) ретінде іске қосылады. Бұл функция Node.js ішіндегі шаблондық жолдарды (Template Literals) пайдаланып, тапсырыс туралы мәліметтерді түсінікті құрылымдалған мәтіндік хабарламаға жинақтайды. 
3) Желілік интеграция және HTTP сұраныс: Хабарлама дайын болғаннан кейін, Axios HTTP-клиенті арқылы ресми Telegram эндпоинтына синхронды POST сұраныс ретінде жіберіледі. Мұндағы ең маңызды қауіпсіздік талабы: құпия кілт пен идентификаторлар бағдарламаның ашық кодында жазылмайды. Олар `dotenv` (`.env`) орта айнымалыларында қауіпсіз сақталады.
4) Қателіктерге төзбеушілік (Fault Tolerance) және асинхронды оқшаулау: Көрсетілген чат бойынша хабарлама (Telegram message) қысқа уақыт аралығында келіп түседі. Алайда, егер желілік байланыс үзілсе немесе Telegram API уақытша қолжетімсіз болса, бұл бағдарламаның негізгі процесін тоқтатпауы керек. Сол себепті жіберу процедурасы қатаң `try/catch` блогы арқылы қауіпсіздендірілген. Бұл архитектуралық шешім жобаның бизнес коммуникациясын кедергілерге төзімді етуіне үлес қосады.

 

Сурет 7. Telegram Bot хабарландыру алгоритмінің блок-схемасы.

Жобаның код базасы көлемді болғандықтан және оның құрамындағы микросервистер бірнеше тәуелсіз каталогтардан (фронтенд, тапсырыстар, өнімдер) тұрғандықтан, жұмыс процесін ұйымдастыру үшін Git үлестірілген нұсқаларды бақылау жүйесі (Version Control System - VCS) пайдаланылды. Барлық жұмыс тарихы мен кодтың жаңару кезеңдері GitHub платформасындағы бұлтты жобалық кеңістікте сақталып отырды [29].
Жобада монорепо (monorepo) архитектуралық үлгісі қолданылды. Бұл барлық сервистер үшін бөлек код базалары ашпай, бәрін бір орталықтан басқаруға мүмкіндік береді. Мұндай тәсілдің арқасында бэкенд пен фронтенд өзгерістерін логикалық тұрғыда байланыстырып, ортақ транзакция (commit) ретінде сақтау мүмкіндігі жасалды. Жобаны әзірлеу барысында `main` (өндірістік тармақ) және `dev` (сынақ тармағы) арасындағы интеграциялық қадамдар бірізді жүзеге асты.

3.1 Бағдарламалық жүйені жүзеге асыру

Бағдарламалық жасақтаманың бастапқы коды монорепо (monorepo) қағидасымен жұмыс істейді. Бұл дегеніміз, фронтенд пен бэкенд үшін бөлек-бөлек Git жобалары құрудың орнына, бүкіл жоба бір ғана ортақ түбірлік (root) кеңістікте сақталады. Мұндай тәсіл код базасын басқаруда, нұсқаларды қадағалауда және `docker-compose` арқылы жүйені бір пәрменмен іске қосуда жеңілдік береді. Жүйенің архитектурасы бірнеше тәуелсіз каталогтарға бөлінген, олардың өз міндеттері мен жеке `package.json` тәуелділіктері бар:
- `api-gateway/`: Жүйенің сыртқы әлеммен байланысатын бірінші және ең басты «қақпасы». Бұл қызмет Express.js микро-фреймворкі негізінде жасалған прокси-сервер болып табылады. Оның ішінде негізгі `server.js` файлы орналасқан, онда `http-proxy-middleware` арқылы барлық бағыттау ережелері (routing rules) жазылған. Мысалы, егер сұраныс `/api/orders` адресіне келсе, ол ішкі желідегі `orders-service` порттарына бағытталады. Сондай-ақ, осы каталогта CORS (домен аралық ресурс бөлісу) саясаттарының баптаулары мен трафикті шектеу (rate limiting) логикасы бағдарламаланған. Бұл әдіс хакерлердің DDoS шабуылдары кезінде жүйенің шамадан тыс жүктелуіне жол бермейді және барлық ішкі микросервистер үшін қауіпсіздік қалқанын қамтамасыз етеді.
- `orders-service/`: Бұл «Stolyarniy dvor» жобасының басты логикалық және бизнес-есептеу орталығы. Папканың ішкі құрылымы классикалық «MVC» (Model-View-Controller) немесе қабатталған архитектура (Layered Architecture) үлгісін ұстанады. Мұнда:
- `prisma/` каталогы деректер базасының миграциялық тарихын және Prisma декларативті схемасын (`schema.prisma`) сақтайды.
- `routes/` папкасы тек қана HTTP интерфейстерін (эндпоинттерді) жариялауға жауапты, яғни қандай URL қандай функцияны шақыратынын анықтайды.
- `controllers/` бөлімі клиенттен келген мәліметтерді қабылдап, олардың дұрыстығын тексереді және клиентке нақты жауаптар (HTTP 200 OK немесе 400 Bad Request) қайтарады.
- `services/` папкасында ең ауыр іскерлік логика жатыр (мәселен, жалпы соманы есептеу, аналитикалық есептер шығару). Сондай-ақ Telegram хабарламаларын жіберетін оқшауланған асинхронды файлдар осы бөлімде орналасқан. Мұндай бөлу кодты модульді (modular) етеді.
- `products-service/`: Жекеменшік деректер базасы бар, тек қана өнімдер каталогын (есіктер, шкафтар, үстелдер) басқаруды жүзеге асыратын шағын микросервис. Оның негізгі міндеті-тауарлар құрылымын, оларды оқу (fetch) мен сақтау логикасын реттеу. Бұл сервис тәуелсіз түрде жұмыс істейді және клиент немесе `orders-service` сұраған кезде JSON форматында тауар сипаттамаларын, бағаларын және суреттерін жіберіп тұрады. Бұл бағдарламаның «өнімдік дисплейі» қызметін атқарады.
- `frontend/`: React және Vite құрастырушысы (bundler) негізінде жазылған барлық пайдаланушы интерфейсінің (веб-сайттың) коды. Фронтенд код базасы өз ішінде өте тығыз және қатаң тәртіпке бағынған иерархиядан тұрады:
- `src/components/`: Қайта қолданылатын «ақылсыз» (dumb) немесе әмбебап UI-элементтер. Мысалы, Түймелер (Buttons), Инпуттар (Inputs), Модальді терезелер (Modals), Карточкалар (ProductCards). Бұлар өзіне деректерді сырттан (props) алады.
- `src/pages/`: Басты бет, пайдаланушы профилі, админ панелі, аутентификация (login/register) сияқты бизнес локикасы байқалатын тұтас экрандар (views).
- `src/services/` (немесе `api/`): Бұл каталог бэкендпен байланысатын жалғыз көпір. Себебі барлық серверге баратын Axios API сұраныстары кодтың әр жерінде шашыраңқы жатпай, осы жерде орталықтандырылған. Егер порт немесе домен ауысса, әзірлеуші тек осы файлдарға ғана өзгеріс енгізеді.
- `src/store/`: Жаһандық (global) күй тасымалдаушысы ретінде Zustand менеджері қолданылады. Мұнда себеттің (cart) күйі сақталады. Яғни, клиент басты бетте тұрып тауарды себетке салса, Zustand оны жадында сақтап, клиент кез-келген басқа бетке өтсе де, себеттің дұрыс санын көрсетіп тұрады.
Бұл жауапкершіліктері толық ажыратылған бағдарламалық құрылым жүйеге тұрақтылық (stability) береді. Егер тапсырыстар жүйесі қатеге ұрынса, әзірлеушілер тек `orders-service` каталогына шоғырланып, `controllers` мен `services` логикасын тексеріп, қатені тез тауып түзете алады. Жүйенің қалған бөліктері (фронтенд немесе products-service) бұл кезде тоқтамайды. Бұл өндірістік стандарттағы архитектуралық тәсіл (best practice) болып табылады.

3.2 Жүйені тестілеу нәтижелерін талдау

Жүйені өндіріске (Production) енгізбес бұрын оның жоғары тұрақтылығы мен пайдаланушы деректерінің қауіпсіздігі 100% кепілдендірілуі үшін өнеркәсіптік стандарттарға (Software Testing Life Cycle) сай бірнеше деңгейлі қатаң сынақтар мен тестілеу жұмыстары ұйымдастырылды. 
 Функционалдық және Эмуляциялық тестілеу (Manual/UI & E2E Testing): React негізіндегі веб-интерфейстің барлық компоненттері кросс-браузерлік (Chrome, Safari, Firefox) тексеруден өтті. Мұнда барлық батырмалар, парақтан параққа ауысулар мен ең бастысы транзакциялық формалардың сенімділігі эмуляторлар арқылы және қолмен (manual) жүзден аса рет тексерілді. Ең басты мақсат тұтынушының теріс әрекеттері (Edge Cases) кезінде жүйенің эвристикасын бағалау болды. Мәселен, клиент телефон нөмірінің орнына әріптер енгізсе, өте үлкен көлемдегі текст жазса немесе қажетті форманы міндетті толтырмай (required) бос жіберуге тырысса, клиент жүйесі React Error Boundary арқасында ешқашан «ақ экранға» ұрынып құламайды. Оның орнына Zod схемасы іске қосылып, өте әдемі және сыпайы қызыл түсті интерфейстік ескертулер (Toast Messages немесе Form Error spans) шығарып, клиентке қай ұяшықта қате кеткенін дәл нұсқап көрсететіні (User Experience) мінсіз деңгейде расталды.
Интеграциялық және Қауіпсіздік тестілеуі (Integration & Security Testing): Бұл кезеңде микросервистік архитектураның ең осал тұсы саналатын қызметтердің өзара (сервисаралық) байланысы сыналды. API Gateway шлюзі `Products Service` және `Orders Service` микросервистері арасында трафикті қатесіз бағыттай алатындығы, тіпті бір қызметке жасанды түрде үлкен шабуыл (DOS attack simulation) жасалғанда да екінші қызметтің еш іркіліссіз жұмыс істеп тұратыны дәлелденді (Service Isolation). Сондай-ақ, үшінші тарап (Third-party) API-лермен жұмыс мұқият тексерілді: Telegram Bot-тың хабарламасы нақты 2 секунд ішінде өндіріс чатына келіп түсетіні расталды. Ең бастысы, егер Telegram серверлері өшіп қалса (Timeout) немесе .env ішіндегі API кілті қате болса, Node.js ішіндегі `try/catch` қауіпсіздік блоктарының арқасында бүтіндей сервердің құлап қалмайтындығы (Graceful Error Handling) және басқа тұтынушылардың тапсырыс бере алу мүмкіндігі үзілмейтіні тіркелді. JWT токендердің жарамдылық мерзімі мен жалған токендерді сұрыптау логикасы да қатаң аудиттен өтті.
Өнімділік және жүктеме тестілеуі (Stress & Load Testing): Жобаның жоғары жүктемелерге төзімділігін тексеру үшін Apache JMeter немесе Postman Runner сияқты арнайы жүктеме генератор құралдары пайдаланылды. Шамамен 200-ден 500-ге дейінгі виртуалды пайдаланушы (Virtual Users) бір мезетте серверге ұжымдық (concurrent load) тапсырыс сұранысын жіберген кезде жүйенің реакциясы өлшенді. Нәтижесінде, Prisma ORM біріктірілген пул (Connection Pooling) қосылымдарын оңтайлы басқарып, PostgreSQL дерекқорындағы кестелер (Tables) мен кілттер (Primary Keys) шатаспай, барлық ACID транзакцияларды кезекке дұрыс қойғаны расталды. Node.js платформасының V8 қозғалтқышы өзінің әйгілі асинхронды Event Loop циклі арқылы (I/O non-blocking) мыңдаған сұраныстарды ешқандай іркіліссіз, кезек-кезек жеңіл өңдегені көрінді. Рендеринг пен API-дан қайтатын жауаптардың кешігу уақыты (Latency) орташа максималды жүктемеде 200-300 миллисекундтан аспады. Бұл халықаралық интернет-жаңалықтар мен дүкендер үшін өте жоғары көрсеткіш (High Performance) болып табылады.

Сурет 16. Postman бағдарламасы арқылы API сұранысын тексеру нәтижесі.

Жүйенің ақпараттық қауіпсіздігін растау мақсатында халықаралық OWASP стандарттарына негізделген қатаң тестілеу жұмыстары жүргізілді. Сынақ барысында келесідей шабуыл түрлері эмуляцияланып, қорғаныс механизмдері тексерілді:
Авторизация және Рөлдік қатынасты тексеру: 
Жүйенің рұқсат беру механизмі Postman арқылы тікелей API сұраныстарын жіберу әдісімен тексерілді. Эмуляция барысында жарамдылық мерзімі өтіп кеткен (Expired) немесе жалған сандық қолтаңбасы бар JWT (JSON Web Token) токендері әкімшілік эндпоинттарға (`/api/orders/all`) бағытталды. Тест нәтижесінде `authMiddleware` серверлік қабаты мұндай сұраныстарды сәтті бұғаттап, қатаң түрде `401 Unauthorized` немесе `403 Forbidden` HTTP-статусын қайтарғаны расталды. Құпия сөздерді тексеру барысында дерекқорға жазылған барлық мәліметтердің `bcrypt` алгоритмімен хэштелгені аудит арқылы расталды.
SQL Injection шабуылдарын эмуляциялау: 
Жүйенің мәліметтер базасына рұқсатсыз енуді тексеру үшін іздеу өрістеріне және авторизация формаларына `' OR 1=1 --` сияқты SQL Injection скрипттері енгізілді. Тестілеу барысында `Prisma ORM` абстракциялық қабаты бұл сұраныстарды орындалатын команда ретінде емес, қауіпсіз мәтін ретінде сүзіп, шабуылды толықтай бейтараптандырды. Деректерді санитаризациялауды (Data Sanitization) тексеру үшін `Zod` валидациясына сәйкес келмейтін пішімдер жіберілді, нәтижесінде жүйе сервер деңгейіне дейін жеткізбей сұраныстарды қабылдамады.
CORS конфигурациясы мен Желілік қорғанысты сынау: 
Бөгде домендерден жасалған сұраныстардың блокталуын тексеру үшін CORS (Cross-Origin Resource Sharing) саясаты Postman арқылы бөтен локальді порттардан сыналды. Сервер тек «Ақ тізімге» (Whitelist) енгізілген фронтенд-домендерден ғана сұраныстар қабылдайтыны және рұқсат етілмеген домендерге кіруді сәтті бұғаттайтыны расталды. Қосымша ретінде `helmet` мидлвары орнатқан HTTP Headers параметрлері тексеріліп, кросс-сайттық скриптинг (XSS) қаупінің алдын алғаны актіленді.
Сынақ қорытындысы бойынша жүйенің коммерциялық айналымы мен тұтынушы деректерінің қорғанысы қауіпсіздік талаптарына толық жауап беретіні дәлелденді.

3.3 Жүйенің жұмыс тиімділігін арттыру және бұлттық ортаға орналастыру

Бағдарламалық платформаның тәулік бойы үздіксіз жұмыс істеуін (High Availability) қамтамасыз ету үшін қосымша Docker контейнерлік ортасына орналастырылды. Контейнерлеу технологиясы бағдарлама кодын, оған қажетті кітапханалар мен баптауларды бір оқшауланған құрылымға біріктіреді. Бұл шешім серверді көшіру немесе жүйені жаңарту кезінде туындайтын конфигурациялық қателіктердің алдын алып, ортаның тұрақтылығын сақтайды.
Инфрақұрылым `docker-compose.yml` скриптісі арқылы ұйымдастырылды. Дерекқор қабаты `postgres:16-alpine` бейнесіне негізделсе, әрбір микросервис (API Gateway, Orders, Products) және React клиенті жеке Node.js контейнерлерінде іске қосылды [20]. Бұл контейнерлер ішкі виртуалды желі (Docker Network) арқылы байланысып, қауіпсіз деректер алмасуды жүзеге асырады.
Қолданбаны өндірістік ортаға шығару үшін Render бұлтты платформасы (PaaS) қолданылды. Платформа GitHub репозиторийімен тікелей байланысып, үздіксіз интеграция мен жеткізу (CI/CD) құбырын қамтамасыз етті. Бастапқы код жаңартылған кезде жүйе веб-хук (webhook) сигналдары арқылы автоматты түрде құрастырылып, нөлдік іркіліспен (Zero-downtime deployment) жаңартылады. Клиенттік трафик Nginx (Reverse Proxy) сервері арқылы жаңа нұсқаға бағытталады, сондай-ақ платформа автоматты түрде SSL/TLS қауіпсіздік сертификаттарын орнатады.
Жоғарыда аталған сәулеттік шешімдердің негізінде кәсіпорын платформасы жоғары өнімділікке қол жеткізді, ал тұтынушылар жүйені кедергісіз әрі тұрақты түрде пайдалану мүмкіндігіне ие болды.

ҚОРЫТЫНДЫ

Дипломдық жұмыс барысында «Stolyarniy dvor» ағаш өңдеу цехына арналған кешенді микросервистік веб-қосымша әзірленіп, сәтті іске асырылды. Жұмыстың мақсаты онлайн дүкен құру ғана емес, мекеменің ішкі бизнес-процестерін: тапсырыстар айналымын, бизнес-аналитиканы және корпоративтік менеджментті толығымен цифрландыратын платформа жасау болды.
Бірінші тарауда ағаш өңдеу өнеркәсібіндегі дәстүрлі тапсырыс қабылдау проблемалары зерттелді. Қолмен есептеу мен қағаз жүзінде жүргізілетін ескілікті басқару процестеріндегі ақпараттың жоғалу қаупі, адам факторымен байланысты қателіктер және аналитиканың жоқтығы негізге алынып, дайын коммерциялық шешімдерге (Shopify, AmoCRM және т.б.) салыстырмалы талдау жасалды. Нарықтағы бос орынды толтыру бағытында заманауи жеке микросервистік шешімнің артықшылықтары далелденді.
Екінші тарауда жобаның іргелі технологиялық архитектурасы жобаланды. Бағдарлама дәстүрлі монолит емес, ауқымды жүктемелерге төтеп беретін және дербес масштабталатын API Gateway, Orders-Service, Products-Service сегменттерінен құралды. React және Vite көмегімен фронтенд жазылса, сервері Node.js және Express фреймворктерінің негізінде жасалды. Деректерді басқару PostgreSQL дерекқорына және Prisma ORM жүйесіне жүктелді. Сонымен қатар, графикалық аналитикаға арналған Recharts және Excel есептерін автоматтандыру логикасы алгоритмделді. Жедел байланыс үшін Telegram Bot API интеграциясының алгоритмі түсіндірілді.
Үшінші тарауда тәжірибелік-бағдарламалық нәтижелер, яғни интерфейс дизайндары, маршрутизация және әкімшілік аналитика тақталарының жұмысы іс жүзінде көрсетілді. Жүйе функционалдық, қауіпсіздік және өнімділік тестілерінен сәтті өтті. Бағдарламалық қамтамасыз ету Docker контейнерлеріне салынып, GitHub арқылы басқарыла отырып, Render бұлтты платформасына толыққанды іске қосылды.
Қорыта келе, «Stolyarniy dvor» цехы қолданбаны іске асыру арқылы тапсырыстарды қабылдау мен басқару тиімділігін ондаған есеге арттыра алатына көз жеткізілді. Кіріс пен шығыстарды цифрлық құралдармен графиктер арқылы талдау, қажет болған кезде бір пернені басу арқылы Excel есебін алу мүмкіндігі бизнестің қаржылық мөлдірлігін кепілдендіреді. Жоба қойылған барлық міндеттерге толықтай жетіп, ғылыми-өндірістік мақсатты орындады.
