# Development Plan

На основе текущего состояния приложения и обнаруженных проблем, здесь представлен пошаговый план дальнейшей разработки.

## Critical fixes
- **[x] Создание страницы `/profile`:** Реализовать заглушку для `ProfilePage.jsx` на фронтенде и пробросить её через роутер `App.jsx`, чтобы редирект для пользователей с ролью `USER` работал корректно.

## Auth & Security
- **[x] Внедрение JWT (Бэкенд):**
  - Подключить `jsonwebtoken` в `orders-service`.
  - Изменить `POST /auth/login`: генерировать токен на основе `userId` и `role` и отправлять его клиенту (в идеале — через `httpOnly` cookie, либо в теле запроса, если фронтенд настроен на это).
  - Сделать middleware `verifyToken` в `orders-service` для защиты приватных роутов.
- **[x] Внедрение JWT (Фронтенд):**
  - Обновить `api.js` для добавления интерсепторов, которые будут прикреплять JWT в `Authorization: Bearer` заголовок, если токен не хранится в куки.
- **Защита маршрутов на фронтенде:**
  - Создать `ProtectedRoute.jsx` компонент.
  - Обернуть роуты `/profile` (для авторизованных) и `/admin` (только для роли `ADMIN`) в `App.jsx`.

## User Features
- **[x] Настройка Профиля:**
  - На странице `ProfilePage` добавить отображение информации о пользователе (имя, телефон).
- **[x] "Мои заказы" (Пользователь):**
  - Эндпоинт на бэкенде: `GET /orders/my` (требует JWT ID для фильтрации своих заказов).
  - На фронтенде (в `/profile`) сделать вывод истории заказов конкретного пользователя, статусов этих заказов (на основе `Order` схемы из базы `orders-service`).
  - [ ] При оформлении заказа через форму (`/custom-order`) отправлять на бэк `userId`, если клиент залогинен.

## Admin Features
- **Запрет доступа к /admin:**
  - Убедиться, что `GET /orders`, `PUT /orders/:id` в `orders-service` защищены middleware на наличие роли `ADMIN`.
- **Защита API Товаров:**
  - Убедиться, что редактирование / добавление / удаление товаров (`products-service`) недоступно обычным юзерам.

## Architecture Improvements
- **Очистка стейта корзины/отзывов при логауте:** Сделать глобальную кнопку `Logout` и функцию очистки хранилища, которая небходима для безопасности при выходе из профиля.
- **Связь Orders/Products:** Из-за микросервисной структуры `productId` в заказах может стать недействительным, если товар удалят. Нужно на бэкенде внедрить проверку через HTTP/API или реализовать логику Soft-Delete (`isDeleted: true`) у товара, чтобы не повреждать историю оформленных ранее заказов.
